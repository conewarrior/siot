## 4. Design System 프로젝트 (9장)

### Page 1: 커버
**"바이브코딩 시대, 디자인 시스템도 자동화할 수 있을까?"**
- 프로세스 마일스톤: 컬러 설계 → 토큰 체계 → Claude Code 자동화
- 핵심 메시지: `/setup-design` 한 줄로 끝나는 디자인 시스템 적용

---

### Page 2: 문제 정의
**"AI가 만든 UI는 왜 일관성이 없을까?"**
- 문제 상황:
  - 하드코딩된 색상값 (#fff, #333)
  - 제각각인 간격 (12px, 15px, 17px)
  - 프로젝트마다 다른 스타일
- 원인: AI에게 규칙을 전달하는 체계가 없음

---

### Page 3: 컬러 시스템 설계
**"WCAG 명도대비율 기반 10단계 스케일"**
- 접근성 기준 충족하는 색상 체계
- Neutral / Primary / Semantic 분류
- 각 스케일의 용도 정의 (배경, 텍스트, 강조 등)

---

### Page 4: 토큰 네이밍 체계
**"색상이 아니라 역할로 부르기"**
- Primitive → Semantic 2단계 구조
- 예시: `gray-500` → `text-secondary`
- 다크모드 대응: 같은 이름, 다른 값

---

### Page 5: CDN + npm 분리 배포
**"토큰은 즉시, 컴포넌트는 버전 관리"**
- `tokens.css` → jsDelivr CDN (즉시 반영)
- `components/` → npm 패키지 (버전 업데이트)
- 분리 이유: 토큰 변경은 빠르게, 컴포넌트는 안정적으로

---

### Page 6: /setup-design 커맨드
**"한 줄 명령어로 디자인 시스템 적용"**
- 실행 내용:
  1. `npm install @geniefy/ui`
  2. `tokens.css` import 추가
  3. `CLAUDE.md`에 규칙 참조 추가
  4. Dependabot 자동 업데이트 설정
- 데모: 터미널 실행 화면

---

### Page 7: design-rules Skill
**"AI가 토큰을 자동으로 사용하게 만들기"**
- UserPromptSubmit Hook으로 UI 키워드 감지
- Generation Protocol 4단계 적용
- 규칙 위반 시 자동 거부/수정
- 결과: 하드코딩 색상 0%

---

### Page 8: 자동 기여 시스템
**"새 컴포넌트가 자동으로 중앙 저장소에"**
- auto-contribute Hook:
  - `components/` 폴더에 파일 생성 감지
  - GitHub API로 자동 커밋
- Dependabot + 자동 머지:
  - 새 버전 배포 시 PR 자동 생성
  - CI 통과 시 자동 머지
- 양방향 동기화 완성

---

### Page 9: 결과 & 확장
**"규칙을 코드로, 코드를 시스템으로"**
- Before/After 비교
- 적용 프로젝트 수, 일관성 개선 지표
- 확장 가능성: 다른 팀, 다른 도구로 확대

## 5. CRO Analysis 프로젝트 (9장)

### Page 1: 커버
**"개발 리소스 없이 랜딩페이지 운영하기"**
- 문제 → 해결 → 시스템 확장의 마일스톤 흐름도
- 핵심 수치: 2시간 → 명령어 한 단어 (몇 초)

---

### Page 2: 시작 - Claude Code로 직접 제작/배포
**"개발자 없이 랜딩페이지 직접 만들기"**
- 상황: 마케팅/기획팀이 랜딩페이지 수정할 때마다 개발팀 의존 (2주 대기)
- 해결: Claude Code로 랜딩페이지 직접 제작 + Vercel 배포
- 결과: 의존성 제거, 즉각적 실험 가능

---

### Page 3: 문제 발견 #1 - 분석에 2시간 소요
**"매번 Beusable 들어가서 분석하는 게 번거롭다"**
- 상황: 히트맵 분석할 때마다 수작업 2시간
- 반복 작업: 스크린샷 → 수치 정리 → 인사이트 도출 → 문서화

---

### Page 4: 해결 #1 - 뷰저블 리포트 자동화
**"Beusable 데이터 → 보고서 자동 생성"**
- 분석 템플릿 설계
- Claude에게 데이터 전달 → 보고서 형태로 정리
- 결과: 2시간 → 10분

---

### Page 5: 적용 - 20기 스터디 랜딩페이지 분석
**"자동화된 리포트로 문제 구간 발견"**
- 스크롤 절벽 구간 발견 (특정 섹션에서 이탈 급증)
- CTA 클릭률 문제 식별
- 정량 데이터 기반 개선 포인트 도출

---

### Page 6: 개선 - 시각적 UI 개선
**"데이터가 말해주는 곳을 고치다"**
- 스크롤 절벽 구간: 콘텐츠 재배치, 시각적 흐름 개선
- CTA 클릭률: 버튼 위치/색상/문구 수정
- Hero → Problem → Transformation 섹션별 개선
- 결과: 이탈률 감소, 클릭률 개선

---

### Page 7: 문제 발견 #2 - 정성적 데이터 부족
**"숫자는 있는데, 왜 이탈하는지 모르겠다"**
- 상황: 히트맵은 "어디서" 이탈하는지 알려주지만 "왜"는 모름
- 필요: 이탈 순간의 사용자 피드백

---

### Page 8: 해결 #2 - Exit Intent 팝업 도입
**"떠나려는 순간, 이유를 묻다"**
- Exit Intent 개념: 마우스가 브라우저 상단으로 이동 시 팝업
- Voiceform 연동: 음성으로 간편하게 피드백 수집
- 뷰저블 트래킹 코드 삽입
- 결과: 가격 부담 39%, 일정 불일치 24.4%, 팀원 상의 12.2%

---

### Page 9: 시스템 확장 & 회고
**"개인 생산성 → 팀 전체가 쓰는 시스템"**
- 매일 아침 메일로 리포트 자동 발송
- 명령어 한 단어로 최신 분석 리포트 생성
- 학습: 문제 발견 → 해결 → 시스템화 패턴의 반복
- 핵심 메시지: 결과물이 아닌 문제 해결 과정의 가치
