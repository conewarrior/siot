디자인 시스템은 더 이상 선택이 아닌 필수가 되었습니다. 패스트캠퍼스에서 디자인 시스템을 처음부터 구축하면서 배운 것들을 공유합니다.

## 왜 디자인 시스템인가

데이원컴퍼니에 합류했을 때, 여러 서비스가 각자 다른 스타일로 만들어져 있었습니다. 같은 버튼인데 서비스마다 다른 색상, 다른 크기, 다른 간격. 개발자와 디자이너 모두 "이게 맞나?" 하는 질문을 반복했습니다.

디자인 시스템은 이런 혼란을 해결합니다. 하지만 단순히 컴포넌트를 모아두는 것만으로는 부족합니다.

## Figma Variables의 등장

2023년 Figma가 Variables 기능을 출시했을 때, 저는 이것이 게임 체인저가 될 것이라 확신했습니다. 기존의 스타일 시스템과 달리, Variables는 진정한 의미의 디자인 토큰을 구현할 수 있게 해줍니다.

### 토큰 구조 설계

저희가 구축한 토큰 구조는 3단계로 이루어져 있습니다:

**1. Primitive 토큰**
```
gray-100: #F5F5F5
gray-200: #E5E5E5
blue-500: #3B82F6
```

**2. Semantic 토큰**
```
background-primary: gray-100
text-secondary: gray-600
interactive-primary: blue-500
```

**3. Component 토큰**
```
button-background: interactive-primary
button-text: white
```

이 구조의 장점은 다크모드 전환이 매우 쉬워진다는 것입니다. Semantic 토큰만 변경하면 전체 시스템이 자동으로 업데이트됩니다.

## WCAG 기준 컬러 시스템

접근성은 타협할 수 없는 부분입니다. WCAG 2.1 AA 기준을 충족하기 위해 모든 텍스트와 배경 조합의 명도 대비를 4.5:1 이상으로 맞췄습니다.

특히 다크모드에서 문제가 많이 발생합니다. 라이트모드에서 잘 보이던 색상이 다크모드에서는 눈에 띄지 않거나, 너무 튀는 경우가 있습니다. 이를 해결하기 위해 각 모드별로 별도의 팔레트를 설계했습니다.

## Storybook 도입

디자인 시스템은 문서화가 생명입니다. 아무리 잘 만들어도 개발자가 찾아서 쓸 수 없으면 의미가 없습니다.

Storybook을 도입하면서 몇 가지 원칙을 세웠습니다:

1. **모든 컴포넌트는 독립적으로 동작해야 한다** — 의존성 없이 단독으로 렌더링 가능
2. **모든 상태를 보여줘야 한다** — hover, active, disabled, error 등
3. **실제 사용 예시를 포함한다** — 코드 스니펫과 함께

## 결과

디자인 시스템 도입 후 체감되는 변화:

- **작업 속도 약 20% 단축** — 매번 새로 만들 필요 없이 조합만 하면 됨
- **디자인 리뷰 시간 감소** — "이 색상 맞아요?" 같은 질문이 사라짐
- **개발 협업 개선** — 명세 문서 대신 Storybook 링크 하나면 충분

## 배운 것들
