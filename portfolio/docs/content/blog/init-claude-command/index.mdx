## 문제: 매번 CLAUDE.md를 처음부터

새 프로젝트를 시작할 때마다 CLAUDE.md를 만들어야 했다. 근데 매번 뭘 넣어야 할지 고민이 됐다. 기술 스택? 빌드 명령어? 그러다 보니 프로젝트마다 형식도 내용도 제각각이었다.

그리고 작업하다 보면 CLAUDE.md가 점점 길어졌다. 온갖 규칙이 추가되다 보니 나중엔 2000자가 넘어갔다. 길어지면 오히려 안 읽히게 된다.

## 원했던 것: 운영 지침 템플릿

처음엔 기술 스택이나 빌드 명령어를 자동으로 분석해서 넣어주는 커맨드를 생각했다. 근데 그건 핵심이 아니었다.

CLAUDE.md의 진짜 목적은 **Claude와 협업할 때의 운영 지침**이다:
- 문서 자체의 관리 규칙 (길이 제한 등)
- 코드 작성 원칙
- 작업 방식 규칙

이런 건 프로젝트마다 크게 다르지 않다. 기본 템플릿이 있으면 된다.

## 해결: /init-claude 커맨드

`~/.claude/commands/init-claude.md`를 만들었다. 새 프로젝트에서 `/init-claude` 실행하면 기본 CLAUDE.md가 생성된다.

### 기본 템플릿

```markdown
# CLAUDE.md

## 문서 관리 규칙

- 이 파일은 500자 이내 유지 (최대 1500자 초과 금지)
- 새 규칙 추가 시 오래된 규칙 정리
- 프로젝트 진행하며 학습한 패턴만 기록

## 코드 원칙

- 간결하게: 최소한의 코드로 목적 달성
- 재사용 우선: 기존 것 활용, 없으면 재사용 가능하게 작성
- 중복 금지: 비슷한 코드 반복되면 추상화

## 작업 규칙

- 변경 전 기존 코드 먼저 읽기
- 요청받은 범위만 수정, 과도한 리팩토링 금지

## 코드 스타일

- (프로젝트 진행하며 추가)
```

### 핵심 원칙

**1. 짧게 유지 (500자 기본, 1500자 한도)**

길어지면 안 읽힌다. 정말 필요한 규칙만 남기고, 새 규칙 추가할 때 오래된 건 정리한다.

**2. 처음엔 최소한으로**

빈 칸으로 시작하는 섹션이 있다. "코드 스타일"은 프로젝트 진행하면서 필요할 때 추가한다. 미리 채워넣지 않는다.

**3. 진화하는 문서**

처음부터 완벽할 필요 없다. 작업하다가 "이건 규칙으로 남겨야겠다" 싶을 때 추가한다.

[📸 /init-claude 실행 후 생성된 CLAUDE.md]

## 글로벌 커맨드로 등록

`~/.claude/commands/`에 넣어서 어떤 프로젝트에서든 사용 가능하게 했다. 프로젝트별 커맨드가 아니라 글로벌 커맨드다.

```
~/.claude/
└── commands/
    └── init-claude.md
```

## 결과

이제 새 프로젝트 시작할 때:

```bash
cd new-project
claude
> /init-claude
```

기본 운영 지침이 담긴 CLAUDE.md가 생성된다. 거기서 시작해서 프로젝트에 맞게 조금씩 추가하면 된다.

## 배운 점

CLAUDE.md는 프로젝트 문서가 아니라 **협업 규칙서**다. 기술 스택이나 빌드 명령어보다, "어떻게 작업할지"가 더 중요하다.

그리고 짧을수록 좋다. 500자면 충분하다.
