# PRD: /setup - Claude Code 초기 세팅 커맨드

## 1. 개요

### 배경
팀원들이 새 프로젝트를 시작하거나 Claude Code를 처음 사용할 때, 디자인 시스템과 개발 가이드를 일일이 설정해야 하는 번거로움이 있다. 특히 비기획 직군(개발자 포함)은 기획/디자인 관련 설정에 익숙하지 않아 일관성 없는 환경이 만들어지기 쉽다.

### 문제 정의
- 팀마다, 사람마다 다른 CLAUDE.md 설정
- 디자인 시스템 토큰/컴포넌트 참조 방법 모름
- 폴더 구조, 네이밍 컨벤션 불일치
- 매번 수동 설정 → 시간 낭비 + 실수 발생

### 해결 방안
**단일 커맨드(`/setup`)로 일관된 개발 환경 자동 구성**

---

## 2. 목표

### 핵심 목표
1. `/setup` 한 번으로 Claude Code 개발 환경 완성
2. 팀 전체가 동일한 디자인 시스템과 개발 규칙 사용
3. 비개발자도 쉽게 시작할 수 있는 진입장벽 낮추기

### 성공 지표
- 신규 팀원이 프로젝트 시작까지 걸리는 시간 단축
- CLAUDE.md 일관성 확보 (팀 내 동일한 규칙)
- 디자인 토큰 사용률 증가

---

## 3. 타겟 사용자

### 주요 대상
- **개발자**: 기획 경험 없이 코드만 쓰던 개발자
- **비개발 직군**: 디자이너, PM, 마케터 등 Claude Code 입문자

### 사용자 페르소나
> "디자인 시스템이 있다는 건 알지만, 어디서 어떻게 가져다 쓰는지 모르겠다.
> 매번 선임한테 물어보기도 눈치 보인다."

### 사용 시나리오
- 각자가 새 프로젝트/브랜치에서 `/setup` 실행
- 공용 DS 저장소는 이미 팀에 존재 (별도 구축 필요)

## 5. 기술 결정 사항

### 의사결정 기록

#### Q1. 디자인 토큰 관리 방식?
**결정: Tailwind + CSS Variables 조합 (shadcn/ui 스타일)**

| 고려한 옵션 | 선택 여부 | 이유 |
|------------|---------|------|
| 순수 Tailwind config | X | 런타임 변경 불가 |
| 순수 CSS Variables | X | Tailwind 생태계 활용 어려움 |
| **Tailwind + CSS Vars** | **O** | 두 장점 모두 취함, shadcn이 검증 |

#### Q2. 컴포넌트 공유 방식?
**결정: 토큰 CDN + 컴포넌트 npm 패키지**

| 고려한 옵션 | 선택 여부 | 이유 |
|------------|---------|------|
| shadcn 복사 방식 | X | 단일 소스 유지 불가, 업데이트 수동 |
| 전체 npm 패키지 | X | 토큰 변경 시 매번 업데이트 필요 |
| **토큰 CDN + 컴포넌트 npm** | **O** | 토큰은 실시간 반영, 컴포넌트는 버전 관리 |

- **토큰**: jsDelivr CDN으로 실시간 동기화
- **컴포넌트**: npm 패키지로 버전 관리

#### Q3. 폴더 구조?
**결정: Next.js App Router 기본 구조**

| 고려한 옵션 | 선택 여부 | 이유 |
|------------|---------|------|
| Feature-based | X | 초기 설계 복잡, 비개발자에게 어려움 |
| **Next.js 기본** | **O** | 공식 문서와 일치, 진입장벽 낮음 |

#### Q4. DS 저장소 위치 설정?
**결정: 하드코딩 + fallback 입력**

- 기본값: 회사 DS 저장소 URL 하드코딩
- 다른 DS 사용 시: 실행 시 입력 가능

## 6. /setup 커맨드 동작

### 실행 순서
```
1. CLAUDE.md 기본 템플릿 생성 (/init-claude 내용)
2. 디자인 시스템 관련 스킬/에이전트 설치
3. CLAUDE.md에 DS 규칙 추가
4. 완료 메시지 및 다음 단계 안내
```

### CLAUDE.md에 추가될 내용

```markdown
## 디자인 시스템

### 토큰 참조
- CSS Variables로 정의된 토큰 사용
- 직접 색상값/크기값 하드코딩 금지
- 예: `bg-primary` (O), `bg-[#3b82f6]` (X)

### 컴포넌트 규칙
- @회사명/ui 패키지의 컴포넌트 우선 사용
- 새 컴포넌트는 기존 토큰 기반으로 작성
- 컴포넌트 위치: src/components/

### 폴더 구조
```
src/
├── app/          # Next.js App Router 페이지
├── components/   # 공유 컴포넌트
│   └── ui/       # 기본 UI 컴포넌트
├── lib/          # 유틸리티 함수
└── styles/       # 전역 스타일
```

### 네이밍 컨벤션
- 컴포넌트: PascalCase (Button.tsx)
- 파일: kebab-case (user-profile.ts)
- CSS 클래스: Tailwind 유틸리티 사용
```

## 부록: 인터뷰 주요 논의

### 토큰 관리 방식 논의
> "tailwind로 작업하면 컴포넌트 생김새에 제약이 생기지 않나? 토큰값도 바꾸기 어려운 건 아닌가?"

→ Tailwind는 `tailwind.config.js`에서 토큰을 정의하면 전체 컴포넌트에 일괄 적용됨. CSS Variables와 조합하면 런타임 변경도 가능.

### 컴포넌트 공유 방식 논의
> "복사 자체가 직관적이지 않다. 단일 소스를 바라본다면 업데이트가 자동 반영되어야 한다."

→ npm 패키지만으로는 자동 업데이트 안 됨. **토큰은 CDN으로 실시간 동기화**, **컴포넌트는 npm으로 버전 관리** 방식으로 해결.

### CDN 선택 논의
> "토큰 CDN은 어떤 방식으로 만들어져서 어디로 가게 되는 거지?"

→ jsDelivr 사용. GitHub에 푸시하면 자동으로 CDN에 반영됨.
- 예: `https://cdn.jsdelivr.net/gh/회사명/design-system/tokens.css`
