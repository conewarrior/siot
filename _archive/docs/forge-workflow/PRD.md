# PRD: Forge 워크플로우 자동화

## 개요

회의록이나 초안 아이디어에서 시작해 PRD 문서를 생성하고, 심층 질문을 통해 구체화한 뒤, 프로젝트 구조화 및 Linear 이슈 생성, 병렬 개발, 리뷰까지 자동화하는 워크플로우 시스템.

**핵심 가치:** 아이디어를 실행 가능한 프로젝트로 단조(forge)하는 전체 과정을 자동화

## 워크플로우 구조

```
┌─────────────────────────────────────────────────────────────────┐
│  /forge [원천파일.md]                                            │
│                                                                 │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ 1. PRD 초안 생성 (inline 지침)                            │   │
│  │    회의록/아이디어 → PRD 초안                             │   │
│  └──────────────────────────────────────────────────────────┘   │
│                           ↓                                     │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ 2. /ask-question (기존 커맨드)                            │   │
│  │    PRD 심층 질문 → 구체화                                 │   │
│  └──────────────────────────────────────────────────────────┘   │
│                           ↓                                     │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ 3. architect (에이전트)                                   │   │
│  │    ├─ PRD → Project Description 템플릿                   │   │
│  │    ├─ 마일스톤/이슈 분리                                  │   │
│  │    ├─ 각 이슈별 DEV.md 생성                              │   │
│  │    ├─ 병렬 가능 구간 분석 & 마킹                          │   │
│  │    └─ Linear 프로젝트/이슈 생성                           │   │
│  └──────────────────────────────────────────────────────────┘   │
│                           ↓                                     │
│              --full 옵션 시 아래 단계 자동 실행                   │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ 4. 병렬 개발 실행 (Task 직접 호출)                         │   │
│  │    병렬 가능 이슈 → 동시 Task 실행                        │   │
│  │    순차 이슈 → 순서대로 Task 실행                         │   │
│  └──────────────────────────────────────────────────────────┘   │
│                           ↓                                     │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ 5. reviewer (기존 에이전트)                               │   │
│  │    코드 리뷰 & 수정                                       │   │
│  └──────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘


┌─────────────────────────────────────────────────────────────────┐
│  /wrap                                                          │
│                                                                 │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ Linear 동기화 (inline 로직)                               │   │
│  │    ├─ 이슈 상태 변경 (Done 등)                           │   │
│  │    ├─ 구현 내용 코멘트 추가                               │   │
│  │    └─ 체크리스트 업데이트                                 │   │
│  └──────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

## 구성 요소

### 커맨드

| 이름 | 역할 | 상태 |
|------|------|------|
| `/forge` | 전체 워크플로우 트리거 (기획 → 구조화 → 개발 → 리뷰) | 신규 |
| `/wrap` | 작업 마무리 및 Linear 동기화 | 신규 |
| `/ask-question` | PRD 심층 질문으로 구체화 | 기존 |

### 에이전트

| 이름 | 역할 | 상태 |
|------|------|------|
| `architect` | PRD → 프로젝트 구조화, 이슈 분리, 병렬 분석, Linear 생성 | 신규 |
| `reviewer` | 코드 리뷰 및 수정 | 기존 |

## 출력 구조

```
/Project
├── 원천소스.md                    # 입력: 회의록 또는 초안 아이디어
├── Project-PRD.md                # architect 출력
│     ├─ Project Description      # 프로젝트 개요 (Linear Description용)
│     ├─ 마일스톤 목록             # 일정별 마일스톤
│     └─ 이슈 목록                 # 병렬 가능 구간 마킹 포함
└── /Issues
    ├── ISSUE-001.md              # 이슈 상세 (Linear 이슈 템플릿)
    ├── ISSUE-001-DEV.md          # 구현 가이드 (개발용)
    ├── ISSUE-002.md
    ├── ISSUE-002-DEV.md
    └── ...
```

## 커맨드 상세

### /forge

**사용법:**
```bash
/forge [원천파일.md]              # 기본: PRD + 질문 + 구조화까지
/forge [원천파일.md] --full       # 풀사이클: 개발 + 리뷰까지 자동
/forge [원천파일.md] --no-linear  # Linear 연동 없이 로컬만 생성
```

**실행 프로세스:**

1. **PRD 초안 생성** (inline)
   - 원천 소스(회의록/아이디어)를 읽고 PRD 초안 작성
   - 핵심 목표, 문제점, 해결 방향 추출

2. **/ask-question 호출**
   - 생성된 PRD 초안을 기반으로 심층 질문 시작
   - 모호한 부분, 누락된 요구사항, 기술적 결정 사항 확인
   - 질문 완료 후 PRD 보완

3. **architect 에이전트 호출**
   - 완성된 PRD를 프로젝트 템플릿으로 변환
   - 마일스톤/이슈 분리
   - 각 이슈별 DEV.md 생성
   - 병렬 가능 구간 분석 및 마킹
   - Linear 프로젝트/이슈 생성 (--no-linear 아닌 경우)

4. **병렬 개발 실행** (--full 옵션 시)
   - architect가 마킹한 병렬 구간 확인
   - 병렬 가능 이슈들은 동시에 Task 실행
   - 순차 이슈들은 순서대로 Task 실행
   - 각 Task는 해당 이슈의 DEV.md를 참조

5. **reviewer 호출** (--full 옵션 시)
   - 전체 코드 리뷰
   - 문제점 수정

### /wrap

**사용법:**
```bash
/wrap                            # 현재 프로젝트 이슈들 Linear 동기화
/wrap --project [프로젝트명]      # 특정 프로젝트 지정
```

**실행 프로세스:**

1. 프로젝트 폴더의 이슈 목록 확인
2. 각 이슈별 구현 상태 파악 (완료/진행중/미시작)
3. Linear 이슈 상태 업데이트
4. 구현 내용 코멘트 추가
5. 체크리스트 업데이트

## architect 에이전트 상세

### 입력
- 완성된 PRD 문서

### 출력
- Project-PRD.md (프로젝트 설명 + 마일스톤 + 이슈 목록)
- Issues 폴더 (개별 이슈 문서 + DEV 문서)
- Linear 프로젝트/이슈 (옵션)

### 핵심 기능

**1. PRD → 프로젝트 템플릿 변환**
- 프로젝트 Description 생성 (Linear Description용)
- 목표, 범위, 기술 스택 정리

**2. 마일스톤/이슈 분리**
- PRD에서 작업 단위 추출
- 마일스톤 (큰 단위) → 이슈 (작은 단위) 계층화
- 각 이슈에 예상 작업 범위 명시

**3. 병렬 가능 구간 분석**
- 이슈 간 의존성 분석
- 파일 충돌 가능성 분석
- 병렬 실행 가능한 이슈 그룹 마킹

```markdown
<!-- PARALLEL_START -->
- ISSUE-001: 컴포넌트 A 구현
- ISSUE-002: 컴포넌트 B 구현
- ISSUE-003: API 엔드포인트 구현
<!-- PARALLEL_END -->

<!-- SEQUENTIAL -->
- ISSUE-004: 통합 테스트 (위 3개 완료 후)
<!-- SEQUENTIAL_END -->
```

**4. DEV.md 생성**
- 각 이슈별 구현 가이드
- 수정 가능한 파일 범위 (scope)
- 구현 체크리스트
- 참고 코드/패턴

**5. Linear 연동**
- Linear MCP를 통한 프로젝트 생성
- 이슈 생성 (템플릿 기반)
- 마일스톤 설정

## 병렬 분석 기준

### 의존성 분석
- 이슈 A의 출력이 이슈 B의 입력인 경우 → 순차
- 공통 모듈을 먼저 만들어야 하는 경우 → 순차
- 서로 독립적인 기능인 경우 → 병렬

### 파일 충돌 분석
- 같은 파일을 수정하는 이슈들 → 순차
- 다른 파일을 수정하는 이슈들 → 병렬
- 새 파일만 생성하는 이슈들 → 병렬

- 문제점 3

## 3. 가설 검증 계획

- 작업 항목 2

### 3.2 [작업 트랙 2]
- 작업 항목 1
- 작업 항목 2

- [ ] 완료 기준 1
- [ ] 완료 기준 2
- [ ] 완료 기준 3

## 완료 기준

1. `/forge [파일]` 실행 시 PRD → 질문 → 구조화 완료
2. `--full` 옵션 시 개발 + 리뷰까지 자동 실행
3. Linear에 프로젝트/이슈 정상 생성
4. `/wrap` 실행 시 Linear 상태 동기화 완료
5. 병렬 가능 이슈들이 동시 실행되어 시간 단축

## 고려 사항

### 완전 자동화 시 주의점
- PRD 초안이 모호하면 잘못된 방향으로 개발될 수 있음
- 병렬 작업 중 한 이슈 실패 시 다른 이슈에 영향 가능
- reviewer 단계에서 대부분 잡히지만, 큰 방향 수정은 비용이 큼

### 권장 사용 패턴
1. 처음에는 `--full` 없이 구조화까지만 실행
2. 생성된 이슈들 검토 후 개별 개발 또는 `--full` 재실행
3. 익숙해지면 `--full`로 한번에 실행

- Linear MCP 연결 필수
- 기존 커맨드: `/ask-question`
- 기존 에이전트: `reviewer`
