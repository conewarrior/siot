## 반복되는 기획 → 개발 과정

새 기능을 만들 때마다 같은 과정을 반복했다.

1. 슬랙에서 논의한 내용을 PRD로 정리
2. PRD를 보고 이슈 단위로 분리
3. 각 이슈를 Linear에 등록
4. 개발 시작

특히 이슈 분리가 번거로웠다. "이 태스크는 저 태스크랑 같이 해도 되나?", "파일이 겹치나?" 매번 고민해야 했다.

회의록 하나 넣으면 이슈까지 뽑아주는 커맨드가 있으면 좋겠다고 생각했다.

## 처음 시도: /plan-project

처음에는 `/plan-project`라는 이름으로 시작했다.

```bash
/plan-project meeting-notes.md
```

그런데 이름이 어색했다. "plan"이라고 해놓고 개발까지 하면 이상하다. 이름과 기능이 맞지 않았다.

## 네이밍: 전체 사이클을 담는 이름 찾기

여러 후보를 검토했다.

| 이름 | 뉘앙스 | 탈락 이유 |
|------|--------|----------|
| `/plan-project` | 기획만 하는 느낌 | 개발까지 하면 안 맞음 |
| `/build` | 개발만 하는 느낌 | 기획 과정이 빠져 보임 |
| `/bootstrap` | 초기 세팅 느낌 | 기존 프로젝트에도 쓰니까 |
| `/forge` | 원재료를 단조해서 완성품으로 | 채택 |

**forge**가 적절했다. "날것의 아이디어를 쓸모있는 프로젝트로 단조한다"는 의미가 맞았다.

마무리 커맨드도 필요했다. `/quit`은 무미건조해서 `/wrap`으로 정했다. 작업을 마무리한다는 뜻이다.

## 설계: 커맨드 vs 에이전트 분리

처음에는 전부 커맨드로 만들려고 했다. 그런데 PRD → 이슈 분리 로직이 너무 복잡했다.

분리 기준을 정했다:

| 구분 | 역할 | 예시 |
|------|------|------|
| **커맨드** | 트리거 + 오케스트레이션 | `/forge`, `/wrap` |
| **에이전트** | 복잡한 로직 처리 | `architect` |

커맨드는 "어떤 순서로 뭘 호출할지"만 알고 있다. 실제 복잡한 작업은 에이전트가 한다.

```
/forge [파일]
  ├─► PRD 초안 생성 (inline 로직)
  ├─► /ask-question 호출 (기존 커맨드)
  ├─► architect 에이전트 호출
  └─► 병렬 개발 + reviewer (--full 옵션 시)
```

architect 에이전트는 PRD를 읽고 이슈를 분리하고, 각 이슈별 DEV.md를 생성하고, Linear에 등록한다. 이 로직이 길어서 에이전트로 분리했다.

## architect의 핵심: 병렬 가능 여부 판단

architect 에이전트의 핵심 기능은 "이 이슈들 동시에 해도 되나?" 판단하는 것이다.

두 가지 기준을 정했다:

```markdown
의존성 분석:
- 이슈 A의 출력이 이슈 B의 입력 → 순차
- 공통 모듈 먼저 필요 → 순차
- 서로 독립적 → 병렬

파일 충돌 분석:
- 같은 파일 수정 → 순차
- 다른 파일 수정 → 병렬
- 새 파일만 생성 → 병렬
```

결과물에 이렇게 마킹된다:

```markdown
<!-- PARALLEL_START -->
- ISSUE-001: 컴포넌트 A 구현
- ISSUE-002: 컴포넌트 B 구현
<!-- PARALLEL_END -->

<!-- SEQUENTIAL -->
- ISSUE-003: 통합 테스트 (위 2개 완료 후)
<!-- SEQUENTIAL_END -->
```

`--full` 옵션을 주면 이 마킹을 보고 병렬 개발을 자동으로 실행한다.

## 최종 구조

```
┌─────────────────────────────────────────────┐
│  /forge [파일.md]                            │
│                                             │
│  1. PRD 초안 생성 (inline)                   │
│  2. /ask-question 호출 → 스펙 구체화         │
│  3. architect 에이전트 호출                  │
│     ├─ Project-PRD.md 생성                  │
│     ├─ Issues/ 폴더에 이슈별 파일 생성       │
│     ├─ 병렬 가능 구간 분석                   │
│     └─ Linear 프로젝트/이슈 등록             │
│  4. 병렬 개발 (--full 옵션 시)               │
│  5. reviewer 호출 (--full 옵션 시)           │
└─────────────────────────────────────────────┘

┌─────────────────────────────────────────────┐
│  /wrap                                       │
│                                             │
│  1. Issues/ 폴더에서 상태 파악               │
│  2. Linear 이슈 상태 동기화                  │
│  3. 완료된 이슈에 구현 내용 코멘트            │
└─────────────────────────────────────────────┘
```

## 사용법

기본 실행 (기획까지):
```bash
/forge meeting-notes.md
```

전체 실행 (개발 + 리뷰까지):
```bash
/forge meeting-notes.md --full
```

작업 마무리:
```bash
/wrap
```

## 배운 것

**커맨드와 에이전트는 역할이 다르다.** 커맨드는 "무엇을, 어떤 순서로 호출할지"만 안다. 복잡한 로직은 에이전트로 분리한다. 재사용이 필요하거나 로직이 길면 에이전트다.

**네이밍에 시간을 써야 한다.** `plan-project`에서 `forge`로 바꾸는 데 10분 썼다. 이 10분이 앞으로 수백 번 쓸 때마다 "이게 뭐하는 커맨드지?" 혼란을 줄여준다.

**병렬 가능 여부는 의존성 + 파일 충돌로 판단한다.** "이거 같이 해도 되나?"는 두 가지만 보면 된다. 의존성이 있나? 같은 파일 건드리나? 둘 다 아니면 병렬이다.

**템플릿은 커맨드/에이전트 파일에 inline으로 넣는다.** 별도 파일로 분리하면 공유할 때 불편하다. 한 파일에 다 넣으면 복사 한 번으로 끝이다.

## 한계

아직 실제 프로젝트에서 충분히 테스트하지 못했다. Linear MCP 연동이 제대로 작동하는지, 병렬 분석이 정확한지 검증이 필요하다. 복잡한 의존성을 가진 프로젝트에서 병렬 분석이 틀릴 가능성도 있다.
