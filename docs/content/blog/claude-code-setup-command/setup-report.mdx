# /setup-design 기획서 - 리더 승인용

## 한 줄 요약

`/setup-design` 커맨드로 **디자인 제약 규칙 + 토큰 + 컴포넌트 패키지**를 자동 세팅하고, UI 작업 시 규칙 기반으로 일관된 디자인을 생성하는 것.

---

## 프로젝트 배경

### 현재 문제
- 바이브코딩 시 LLM이 매번 다른 스타일의 컴포넌트 생성
- "예쁜 버튼"이 매번 다르게 해석됨 → 일관성 붕괴
- 디자인 기준이 없어서 사람마다 다른 결과물

### 해결 방향: "디자인을 언어화해서 조직에 흡수하는 실험"

Figma 없이 md 파일만으로 디자인 규칙을 운영하는 방식. 해외에서도 GOV.UK, GitHub Primer 등이 문서 중심 디자인 시스템을 운영하고 있으나, 우리 접근은 더 급진적:
- 디자인 산출물 자체를 제거
- 디자인 사고를 md로 고정
- LLM이 이해하는 **"제약 언어"**로 작성

### 왜 이 방식이 유효한가

> "LLM은 미학을 이해하지 못하고 제약만 이해한다"
> "자유도가 높을수록 불안정, 제약이 강할수록 우아"

1인 1프로덕트 + 바이브코딩 환경에서는:
- 핸드오프 없음
- 빠른 생성 필요
- 강한 제약 → 결과 품질 상승

## 기대 효과

| 항목 | Before | After |
|------|--------|-------|
| 디자인 일관성 | 사람마다 다름 | 규칙 기반 통일 |
| UI 생성 품질 | 매번 다른 스타일 | 제약 언어로 일관된 결과 |
| 토큰 업데이트 | 각 프로젝트 수동 | CDN으로 즉시 반영 |
| 컴포넌트 공유 | 복사/붙여넣기 | npm 패키지로 버전 관리 |

```
│  │ tokens.css   │  │ components/  │  │ design-rules.md  │   │
│  │ (디자인 토큰) │  │ (공용 컴포넌트)│  │ (LLM 제약 규칙)   │   │

│         │                 │                   │             │

└─────────┼─────────────────┼───────────────────┼─────────────┘
          │                 │                   │
          ▼                 ▼                   ▼
     jsDelivr CDN    GitHub Actions       jsDelivr CDN
     (즉시 반영)      (npm 자동 배포)      (로컬 복사)
          │                 │                   │
          ▼                 ▼                   ▼
┌─────────────────────────────────────────────────────────────┐
│                        각 프로젝트                            │
│                                                             │
│  globals.css          package.json         CLAUDE.md       │
│  └─ @import CDN       └─ @회사/ui          └─ 규칙 참조     │
│                                                             │
│  settings.local.json                                        │
│  └─ hook 설정 (src/components/ 감지 → UI 생성 skill)        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 왜 토큰/컴포넌트/규칙을 분리하는가

| 구분 | 배포 방식 | 이유 |
|------|----------|------|
| **토큰** (색상, 간격) | CDN | 값만 바뀌니까 즉시 반영해도 안전 |
| **컴포넌트** (Button 등) | npm | API 변경 가능성 있어 버전 고정 필요 |
| **규칙** (제약 언어) | CDN + 로컬 | 로컬에 복사하되 CDN에서 업데이트 체크 |

## /setup-design 실행 시 동작

```bash
/setup-design
```

| 단계 | 동작 |
|------|------|
| 1 | design-rules.md 로컬 복사 (CDN에서 다운로드) |
| 2 | CLAUDE.md에 design-rules.md 참조 추가 |
| 3 | globals.css에 토큰 CDN import 추가 |
| 4 | `npm install @회사/ui` 실행 |
| 5 | settings.local.json에 hook 설정 추가 |

### 완료 메시지
```
✅ design-rules.md 복사 완료
📐 CLAUDE.md에 규칙 참조 추가됨
🎨 토큰 CDN 연결됨
📦 @회사/ui 설치됨
🔗 hook 설정 완료 (src/components/ 감지)
```

## design-rules.md 설계 방향

일반 디자인 가이드가 아닌 **"LLM UI 제어 언어"**로 작성. tokens.css에 정의된 토큰값(색상, 간격 등)을 참조하여 규칙이 작동한다.

### 작동하지 않는 방식 (금지)
```markdown
- 주요 색상: 파란색
- 깔끔하고 모던한 UI
- 버튼은 상황에 따라 둥글거나 각질 수 있음
```

### 작동하는 방식 (채택)
```markdown
## UI 제약 조건 (반드시 준수)

- 모든 버튼은 반드시 border-radius: var(--radius-md)를 사용해야 한다. 예외 없음.
- 모든 간격은 반드시 8px 단위만 사용해야 한다 (var(--spacing-*)).
- tokens.css 외부의 새로운 색상을 절대 도입하지 마라.
- 화면당 최대 컴포넌트 수: 7개
- 화면당 최대 색상 수: 3개 (텍스트 제외)

## 생성 프로토콜

1. 화면 목적을 한 문장으로 정의
2. 최소한의 컴포넌트 세트 선택
3. 제약 체크리스트 검증
4. 규칙 위반 시 출력 거부
```

### Generation Protocol이란?

LLM이 UI를 생성할 때 **따라야 하는 단계별 절차**. 규칙을 "알면서도 어기는" 현상을 방지하기 위해 검증 단계를 강제한다.

| 단계 | 설명 |
|------|------|
| 1. 목적 정의 | 화면이 뭘 하는 건지 한 문장으로 정리 |
| 2. 컴포넌트 선택 | 필요한 최소한의 컴포넌트만 선택 |
| 3. 규칙 검증 | 토큰, 간격, 색상 등 제약 체크리스트 확인 |
| 4. 위반 시 거부 | 규칙 위반 발견 시 출력하지 않고 수정 |
