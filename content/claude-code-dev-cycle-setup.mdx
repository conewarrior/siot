## PRD만 쓰면 나머지는 자동으로

Claude Code를 쓰면서 점점 욕심이 생겼다. 코드 작성은 Claude가 잘 해주는데 매번 "이거 해줘, 저거 해줘" 하나씩 시키는 게 번거로웠다.

특히 새 기능을 만들 때 패턴이 반복됐다.

1. PRD(기획 문서) 작성
2. 작업 분해
3. 하나씩 구현 요청
4. 완료되면 코드 리뷰
5. 배운 내용 정리

이 과정을 자동화할 수 없을까 생각했다.

## 서브에이전트 개념

Claude Code 문서를 뒤지다가 서브에이전트(Sub-agent)라는 개념을 발견했다. 간단히 말하면 Claude가 다른 Claude를 부르는 것이다. 복잡한 작업이 들어오면 메인 Claude가 다른 인스턴스에 위임할 수 있다.

```
[메인 Claude] ─── "이거 해줘" ──→ [서브에이전트 Claude]
     ↑                              ↓
     └──── 결과 받아서 통합 ←─────────┘
```

`.claude/agents/` 폴더에 마크다운 파일을 만들어두면 특정 역할을 하는 서브에이전트를 미리 정의해둘 수 있다.

## 4개의 단계로 구성

전체 개발 사이클을 이렇게 구성했다.

![개발 사이클 흐름도](/blog/dev-cycle-flow.svg)

### Phase 1: PRD → Development Guide

`prd-to-dev` 서브에이전트가 PRD를 읽고 구체적인 작업 목록을 만든다. 핵심은 병렬 가능한 작업을 찾아내는 것이다. 인증 모듈과 DB 스키마는 서로 겹치지 않으니까 동시에 작업할 수 있다.

```markdown
<!-- PARALLEL_START -->
### Task 1-A: 인증 모듈
- **범위**: src/auth/*

### Task 1-B: DB 스키마
- **범위**: src/db/*
<!-- PARALLEL_END -->
```

`<!-- PARALLEL_START/END -->` 마커로 병렬 작업 구간을 표시한다.

### Phase 2: 병렬 개발

`parallel-dev` 커맨드가 Development Guide를 파싱해서 병렬 작업들을 동시에 실행한다.

```
Phase 2-1: Starting 3 parallel tasks...
[Task 1-A] 인증 모듈...
[Task 1-B] DB 스키마...
[Task 1-C] API 라우트...
[Task 1-B] 완료
[Task 1-A] 완료
[Task 1-C] 완료
```

각 서브에이전트는 자기 파일 범위만 건드린다. `src/auth/*`를 작업하는 에이전트는 `src/db/*`에 절대 손대지 않는다.

### Phase 3: 자동 리뷰

`reviewer` 서브에이전트가 전체 변경사항을 검토한다.

- 불필요한 import 제거
- 디버그용 console.log 삭제
- 중복 코드 정리
- 보안 취약점 체크

자동으로 고칠 수 있는 건 고치고, 수동 검토가 필요한 건 리포트로 남긴다.

### Phase 4: 학습 명문화

마지막으로 `codify` 서브에이전트가 개발 과정에서 배운 내용을 CLAUDE.md에 기록한다.

- 이 프로젝트에서 효과적이었던 작업 분해 방식
- 반복적으로 발견된 이슈
- 프로젝트 특화 컨벤션

다음에 비슷한 작업을 할 때 Claude가 이 내용을 참고할 수 있다.

## 사용 방법

```bash
vim docs/PRD.md

# 2. 실행
/dev-cycle docs/PRD.md

# 3. 대기
```

`/dev-cycle` 커맨드 하나로 전체가 돌아간다.

`.claude/agents/`에 3개 에이전트, `.claude/commands/`에 2개 커맨드를 넣으면 된다.

| 파일 | 용도 |
|------|------|
| `agents/prd-to-dev.md` | PRD → 작업 분해 |
| `agents/reviewer.md` | 코드 리뷰 |
| `agents/codify.md` | 학습 기록 |
| `commands/dev-cycle.md` | 마스터 커맨드 |
| `commands/parallel-dev.md` | 병렬 실행기 |

전역으로 쓰고 싶으면 `~/.claude/`에 넣으면 된다.

---

주의할 점이 있다. 서브에이전트끼리 같은 파일을 건드리면 충돌이 난다. Development Guide에서 각 태스크의 범위를 겹치지 않게 분리하는 게 핵심이다. 또한 서브에이전트는 별도 인스턴스라서 메인 Claude의 대화 맥락을 모른다. PRD 경로 같은 필요한 정보는 명시적으로 전달해야 한다.

이 시스템의 장점은 내가 할 일이 PRD 작성뿐이라는 것이다. 터미널 여러 개로 수동 관리하던 걸 N개 동시 실행으로 바꾸고, 개발 후 직접 리뷰하던 걸 자동 검토로 바꾸고, 배우고 까먹던 걸 CLAUDE.md에 축적하도록 했다. Claude Code의 서브에이전트 시스템을 제대로 활용하면 단순 코드 생성 도구가 아니라 개발 파이프라인 전체를 자동화하는 도구가 된다.
